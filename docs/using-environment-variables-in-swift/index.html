Since the introduction of Swift Concurrency, NIO has added numerous features to make the interop between NIO’s <code><a href='https://swiftinit.org/docs/swift-nio/niocore/channel'>Channel</a></code> eventing system and Swift’s Concurrency primitives as easy as possible.</p><h3 id='EventLoopFuture/Promise%20bridges'><a href='#EventLoopFuture/Promise%20bridges'>EventLoopFuture/Promise bridges</a></h3><p>The first bridges that NIO introduced added methods on <code><a href='https://swiftinit.org/docs/swift-nio/niocore/eventloopfuture'>EventLoopFuture</a></code> and <code><a href='https://swiftinit.org/docs/swift-nio/niocore/eventlooppromise'>EventLoopPromise</a></code> to enable communication between Concurrency and NIO. These methods are <code><a href='https://swiftinit.org/docs/swift-nio/niocore/eventloopfuture.get'>get</a></code> and <code><a href='https://swiftinit.org/docs/swift-nio/niocore/eventlooppromise.completewithtask(_:)'>completeWithTask(_:)</a></code>.</p><aside class='warning'><h3>Warning</h3><p>The future <code><a href='https://swiftinit.org/docs/swift-nio/niocore/eventloopfuture.get'>get</a></code> method does not support task cancellation.</p></aside><p>Here is a small example of how these work:</p><pre class='snippet'><code class='language-swift'><span class='newline'></span><span class='xk'>let</span> <span class='xv'>eventLoop</span>: <span class='xt'>EventLoop</span><span class='newline'> </span><span class='xk'>let</span> <span class='xv'>promise</span> = <span class='xv'>eventLoop</span>.<span class='xv'>makePromise</span>(<span class='xv'>of</span>: <span href='#Channel%20bridges'>Channel bridges</a></h3><p>The <code><a href='https://swiftinit.org/docs/swift-nio/niocore/eventloopfuture'>EventLoopFuture</a></code> and <code><a href='https://swiftinit.org/docs/swift-nio/niocore/eventlooppromise'>EventLoopPromise</a></code> bridges already allow async code to interact with some of NIO’s types. However, they only work where we have request-response-like interfaces. On the other hand, NIO’s <code><a href='https://swiftinit.org/docs/swift-nio/niocore/channel'>Channel</a></code> type contains a <code><a href='https://swiftinit.org/docs/swift-nio/niocore/channelpipeline'>ChannelPipeline</a></code> which can be roughly described as a bi-directional streaming pipeline. To bridge such a pipeline into Concurrency required new types. Importantly, these types need to uphold the channel’s back pressure and writability guarantees. NIO introduced the <code><a href='https://swiftinit.org/docs/swift-nio/niocore/niothrowingasyncsequenceproducer'>NIOThrowingAsyncSequenceProducer</a></code>, <code><a href='https://swiftinit.org/docs/swift-nio/niocore/nioasyncsequenceproducer'>NIOAsyncSequenceProducer</a></code> and the <code><a href='https://swiftinit.org/docs/swift-nio/niocore/nioasyncchanneloutboundwriter'>NIOAsyncChannelOutboundWriter</a></code> which form the foundation to bridge a <code><a href='https://swiftinit.org/docs/swift-nio/niocore/channel'>Channel</a></code>. On top of these foundational types, NIO provides the <code>NIOAsyncChannel</code> which is used to wrap a <code><a href='https://swiftinit.org/docs/swift-nio/niocore/channel'>Channel</a></code> to produce an interface that can be consumed directly from Swift Concurrency. The following sections cover the details of the foundational types and how the <code>NIOAsyncChannel</code> works.</p><h4>NIOThrowingAsyncSequenceProducer and NIOAsyncSequenceProducer</h4><p>The <code><a href='https://swiftinit.org/docs/swift-nio/niocore/niothrowingasyncsequenceproducer'>NIOThrowingAsyncSequenceProducer</a></code> and <code><a href='https://swiftinit.org/docs/swift-nio/niocore/nioasyncsequenceproducer'>NIOAsyncSequenceProducer</a></code> are asynchronous sequences similar to Swift’s <code>AsyncStream</code>. Their purpose is to provide a back pressured bridge between a synchronous producer and an asynchronous consumer. These types are highly configurable and generic which makes them usable in a lot of places with very good performance; however, at the same time they are not the easiest types to hold. We recommend that you <strong>never</strong> expose them in public API but rather wrap them in your own async sequence.</p><h4>NIOAsyncWriter</h4><p>The <code><a href='https://swiftinit.org/docs/swift-nio/niocore/nioasyncchanneloutboundwriter'>NIOAsyncChannelOutboundWriter</a></code> is used for bridging from an asynchronous producer to a synchronous consumer. It also has back pressure support which allows the consumer to stop the producer by suspending the <code><a href='https://swiftinit.org/docs/swift-nio/niocore/nioasyncwriter.yield(contentsof:)'>yield(contentsOf:)</a></code> method.</p><h4>NIOAsyncChannel</h4><p>The above types are used to bridge both the read and write side of a <code><a href='https://swiftinit.org/docs/swift-nio/niocore/channel'>Channel</a></code> into Swift Concurrency. This can be done by wrapping a <code><a href='https://swiftinit.org/docs/swift-nio/niocore/channel'>Channel</a></code> via the <code>NIOAsyncChannel/init(synchronouslyWrapping:configuration:)</code> initializer. Under the hood, this initializer adds two channel handlers to the end of the channel’s pipeline. These handlers bridge the read and write side of the channel. Additionally, the handlers work together to close the channel once both the reading and the writing have finished.</p><p>This is how you can wrap an existing channel into a <code>NIOAsyncChannel</code>, consume the inbound data and echo it back outbound.</p><pre class='snippet'><code class='language-swift'><span class='newline'></span><span class='xk'>let</span> <span class='xv'>channel</span> = <span class='xo'>...</span><span class='newline'> </span><span class='xk'>let</span> <span class='xv'>asyncChannel</span> = <span class='xk'>try</span> <span class='xv'>NIOAsyncChannel</span>&lt;<span class='xt'>ByteBuffer</span>, <span class='xt'>ByteBuffer</span>&gt;(<span class='xv'>wrappingChannelSynchronously</span>: <span class='xv'>channel</span>)<span class='newline'> </span><span class='xk'>try</span> <span class='xk'>await</span> <span class='xv'>asyncChannel</span>.<span class='xv'>executeThenClose</span> { <span class='xv'>inbound</span>, <span class='xv'>outbound</span> <span class='xk'>in</span><span class='newline'> </span> <span class='xk'>for</span> <span class='xk'>try</span> <span class='xk'>await</span> <span class='xv'>inboundData</span> <span class='xk'>in</span> <span class='xv'>inbound</span> {<span class='newline'> </span> <span class='xk'>try</span> <span class='xk'>await</span> <span class='xv'>outbound</span>.<span class='xv'>write</span>(<span class='xv'>inboundData</span>)<span class='newline'> </span> }<span class='newline'> </span>}</code></pre><p>The above code works nicely; however, you must be very careful at what point you wrap your channel otherwise you might lose some reads. For example your channel might be created by a <code>ServerBootstrap</code> for a new inbound connection. The channel might start to produce reads as soon as it registered its IO which happens after your channel initializer ran. To avoid potentially losing reads the channel must be wrapped before it registered its IO. Another example is when the channel contains a handler that does protocol negotiation. Protocol negotiation handlers are usually waiting for some data to be exchanged before deciding what protocol to chose. Afterwards, they often modify the channel’s pipeline and add the protocol appropriate handlers to it. This is another case where wrapping of the <code>Channel</code> into a <code>NIOAsyncChannel</code> needs to happen at the right time to avoid losing reads.</p><h3 id='Asynchronous%20bootstrap%20methods'><a href='#Asynchronous%20bootstrap%20methods'>Asynchronous bootstrap methods</a></h3><p>NIO offers a multitude of bootstraps. To avoid the above problems and enable a seamless experience when using NIO from Swift Concurrency, the bootstraps gained new generic asynchronous methods.</p><p>The next section is going to focus on how to use the methods to boostrap a TCP server and client.</p><h4>ServerBootstrap</h4><p>The server bootstrap is used to create a new TCP based server. This made it easier to get started; however, it came with some downsides. First, implementing business logic inside channel handlers requires the business logic to also handle all of the invariants that the <code><a href='https://swiftinit.org/docs/swift-nio/niocore/channelhandler'>ChannelHandler</a></code> protocol brings with it. This often requires writing complex state machines. Additionally, the business logic becomes very tied to NIO and hard to port between different systems. Because of the above reasons we recommend to implement your business logic using Swift Concurrency primitives and the <code>NIOAsyncChannel</code> based bootstraps.
